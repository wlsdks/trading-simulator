# Trading Simulator — 제품 기획서 (PRD)

> **한줄요약:** "진짜 돈 걱정 없이 주식을 게임처럼 즐기는 가장 재미있는 가짜 돈 트레이딩 놀이터." 레벨·배지·대회·리더보드로 경쟁하고, 친구와 자랑하고, 시즌마다 새 콘텐츠로 돌아오게 만드는 시장 무관 범용 모의투자(페이퍼 트레이딩) **게임**. (실제 자금·실제 체결이 없는 시뮬레이션.)

> **[성격 고지]** 본 문서는 **모의투자 시뮬레이션 게임의 기획 문서**이며, **실제 투자자문·투자중개가 아니다.** 제품은 실제 자금·실제 체결이 없는 시뮬레이션이며, 표시 시세는 시뮬레이션(또는 향후 지연) 데이터다. 본 문서의 모든 수익화·성과·리더보드 관련 서술은 "모의·엔터테인먼트(재미) 목적"을 전제로 하며, 면책·규제·앱스토어 정책 준수를 항상 고려한다.

## 1. 개요·비전

### 1.1 비전 한 문장
> **"진짜 돈 걱정 없이 주식을 게임처럼 즐기는 가장 재미있는 가짜 돈 트레이딩 놀이터."**

핵심은 **재미 우선(fun-first)** 이다. 가짜 돈으로 부담 없이 사고팔며, **레벨·배지·대회·리더보드로 경쟁하고 친구와 자랑하고 수집하는** 재미로 매일 돌아오게 만든다. 차별화는 학습·훈육이 아니라 **재미·소셜·수집·시즌 콘텐츠의 완성도**다 — 기존 게임형 앱(Stock Wars류)이 콘텐츠·소셜·시즌 루프가 얕아 1~2주면 자멸하는 지점을, 우리는 깊은 게임 콘텐츠로 반대로 간다. (단, 재미 게임이어도 법적 가드레일 4종은 지킨다 — §1.4.)

### 1.2 현재 상태 진단 (코드 실측 기반)
- **MVP 코드 존재:** Expo(React Native)+TypeScript. 마켓(12종목, 2초 랜덤워크 시세), 매수/매도 바텀시트, 포트폴리오(총자산·평가손익·현금/주식비중·보유종목·거래내역), AsyncStorage 단일 로컬 계정(`ts.account.v1`), 시작자산 $100,000.
- **구조적 공백:** 계정·백엔드·인증 없음. 게임화·소셜 레이어 0. 리텐션 후크 부재. 즉 **"트레이딩 코어만 있고 돌아올 이유가 없는" 상태.**
- **회계·엔진 결함(실측):** 실현손익(realized P&L) 미집계로 전량청산 시 손익 0 표시, 수익률 분모가 유저 간 비교 불가, 시세가 기기-로컬 비결정론·세션 간 휘발, 변동성 파라미터 시간축 오정규화, buy/sell 클로저 레이스로 더블탭 시 현금 음수 가능. 이 결함들이 게임화·소셜·리더보드의 **하드 선행 과제**다.

### 1.3 포지셔닝
모의투자 시장은 (1) 교육형(Investopedia, Wall Street Survivor), (2) 실전 온보딩형(Webull Paper, thinkorswim paperMoney, TradingView Paper), (3) 게임형(Best Brokers, Stock Wars류)으로 나뉜다. 우리는 **순수 (3)게임형**에 포지셔닝하되, 기존 게임형이 못 가진 **깊은 재미·소셜·수집·시즌 콘텐츠**로 차별화하고 (2)의 UX 완성도를 벤치마크한다. **브로커가 아닌 것이 오히려 무기**다 — 실계좌 전환 압박 없이 순수하게 재미에만 집중할 수 있고, 규제 청정 영역(추후 B2B 라이선스 옵션)도 열려 있다.

### 1.4 우리가 하지 않는 것 (Non-goals)
- 실제 체결·실계좌 연동·실전 투자 유도 → **규제상 영구 배제.**
- 개인화된 매수 추천·유료 매수 시그널·리딩 → 한국 자문업/미국 RIA 트리거, **배제**(법적 가드레일 ③).
- 실제 현금 상금 유료 대회 → **배제**(복권법 리스크, 법적 가드레일 ①). 대회는 비현금 보상만.
- 결제·수도(T+1/T+2)·미수금·PDT 등 정산 규칙 → **의도된 단순화(명시적 non-goal)** 로 게임 흐름 단순화.
- 무거운 학습 커리큘럼·학습성과 측정·수료 트랙 → **핵심에서 배제.** 팁은 선택적 툴팁 수준만(§9).

## 2. 문제·기회

### 2.1 문제 (사용자 관점)
| ID | 문제 | 근거 |
|---|---|---|
| P1 | **진입장벽** — 실전 투자는 무섭고 증권사 앱은 위압적. "부담 없이 지금 눌러보며 재미로 즐길" 안전한 놀이터가 없다. | thinkorswim 온보딩 가파름(반면교사) |
| P2 | **재미의 부재** — 주식은 그 자체로 짜릿하지만, 진짜 돈이 걸리면 즐길 수가 없고, 가짜 돈 앱은 대개 밋밋하다. 오르내림을 마음껏 즐길 안전한 공간이 비어 있다. | 게임형 수요는 크나 만족스러운 제품 부재 |
| P3 | **얕은 게임 루프** — 기존 게임형 모의앱은 대회·리그·수집·시즌 콘텐츠가 얕아 금방 질린다. | 게임형 앱 콘텐츠 빈약(추정) |
| P4 | **리텐션 부재** — 순수 게임 모의앱은 콘텐츠·루프가 약해 1~2주면 이탈. 파산하면 돌아올 이유 없음. | 국내외 앱 리텐션 루프 취약(추정) |

### 2.2 기회 (시장 관점)
- **빈 자리:** 게임형 앱은 많으나 대부분 콘텐츠·소셜·시즌 루프가 얕다. **깊고 재미있는 주식 게임**이라는 자리가 비어 있다.
- **브로커가 아닌 것이 무기:** 실계좌 전환 압박이 없어 순수 재미에만 집중할 수 있다. Webull·토스는 실계좌 전환이 목적이라 이 순수성을 못 가진다.
- **시장 무관 범용성:** 미국주식·크립토·(추후)한국·다자산을 한 시뮬레이터에서. 크립토 24/7 변동성은 게임 리텐션과 궁합이 좋고 데이터 라이선스도 저렴(거래소 API $0).
- **확장 옵션(청정 매출):** 실제 돈 리스크가 없어, 향후 원할 경우 기관·커뮤니티 시트 라이선스로 앱스토어 수수료·자문 규제를 동시에 우회하는 매출선도 열어둘 수 있다.

### 2.3 업계 표준(위생요인)과 차별점
- **위생요인(없으면 감점):** 시작자금 $100,000(업계 표준, 이미 채택 — 좋은 선택), 시장가/지정가/손절 주문, 포트폴리오 손익 추적, 공개/비공개 리더보드.
- **차별점:** ①30초 게스트 온보딩(즉시성), ②레벨·배지·대회·시즌으로 깊은 게임 진행감, ③대회·리그·소셜 자랑 루프, ④수집·시즌 콘텐츠의 깊이.

## 3. 타겟 페르소나

### 페르소나 A — "첫걸음 지수" (Primary · 획득 엔진)
- **상황:** 김도현, 26세, 취업 2년차. 유튜브 밈은 알지만 실계좌는 무섭다. FOMO는 있으나 원금 잃기가 두렵다.
- **니즈:** "가입 없이 지금 당장 눌러보며 재미로 즐기고 싶다. 부담 없이 사고팔고 싶다."
- **핵심 기능:** 30초 게스트 온보딩 → 보장된 첫 승리 → 레벨·배지 진행감.
- **성공 신호:** D1 리텐션, 온보딩 완주율.

### 페르소나 B — "리그 등반가" (Retention/Viral 엔진)
- **상황:** 이서준, 22세, 대학생. 게임 랭크에 익숙, 친구들과 경쟁·인증샷을 좋아함. 소액 실투자 중이며 "실력"을 증명하고 싶어함.
- **니즈:** "친구들 이겨서 자랑하고, 매일 들어올 이유(랭크·스트릭)가 필요하다."
- **핵심 기능:** 비공개 그룹 대회, 주간 리그(승급/강등), 스트릭, 공유 카드.
- **성공 신호:** K-factor, 주간 리텐션, 대회 참여율.
- **주의(2차효과):** 절대수익률 단일 순위는 몰빵·도박 프레임을 부추겨 앱스토어 사행성 심사 리스크를 키운다 → **리스크조정·복합 스코어를 순위에 병용**해 게이밍·심사 리스크를 완화한다(재미는 유지하되 법적 가드레일 준수, §8.4).

### 페르소나 C — "그룹 관리자" (B2B/커뮤니티 · 수익 안정 옵션)
- **상황:** 박선영, 41세, 투자 동아리·크리에이터 커뮤니티 운영자(또는 사내 동호회 리더). 멤버 30명에게 단체 대회를 열어주고 싶으나 관리 도구가 없다.
- **니즈:** "그룹 단위로 시작자본·기간·규칙을 설정하고, 참가자 성과·랭킹을 대시보드로 보고 싶다."
- **핵심 기능:** 비공개 그룹 대회 생성(파라미터화), 운영자 대시보드, 그룹 리더보드, 커스텀 규칙.
- **성공 신호:** 유료 시트 수, 갱신율(NRR), 그룹당 활성 참가자 수.

## 4. 목표·성공지표 (North Star + KPI)

### 4.1 North Star Metric
> **주간 활성 플레이어 (Weekly Active Players, WAP)**
> = 한 주에 **①거래 세션 1회 이상 OR ②대회·리그 참여 OR ③레벨·배지·수집 진행(1회 이상)** 중 하나라도 한 유저 수 = "이번 주에 재미를 위해 돌아온 사람".

**설계 의도:** 재미 우선 게임에서 성공은 곧 **"얼마나 많은 사람이 즐기러 돌아오는가"** 다. WAP는 거래·경쟁·수집 어느 축으로든 **게임에 참여한 유저**를 세므로, 이 지표가 오르면 재미·리텐션·수익이 함께 정렬된다. 단, 법적 가드레일(사행성 심사)을 위해 **"인당 매매 수"는 별도 안티메트릭**으로 감시한다(§4.2).

### 4.2 지표 위계 — 3계층 (게이트 / 진단 / 안티메트릭)

NSM(WAP)의 위계를 흔들지 않도록 모든 보조 지표를 **3계층**으로 명시 분류한다. 이 위계가 §17 마일스톤·게이트의 판정 기준이 된다.

- **게이트 지표(Gate):** NSM 궤적과 릴리스 통과를 좌우하는 하드 판정 지표. **NSM = WAP 주간 궤적**이 유일한 최상위 게이트이며, 온보딩 게이트(게이트 통과율×첫 매수율)도 여기 속한다.
- **진단 지표(Diagnostic):** 성장·수익·참여를 **진단**하되 그 자체를 목표로 최적화하지 않는 지표 — D1/D7/D30 리텐션, 스트릭 유지, 대회 참여율, K-factor, 전환·RLTV. **리텐션·K-factor·ARPU 등은 전부 '검증 목표(가설)'로 일관 라벨링**하며, 마일스톤에서 하드 게이트가 아니라 추세로 관리한다.
- **안티메트릭(Anti-metric):** 오르면 오히려 경보인 이상감지용 지표 — 인당 일평균 매매 수, 하이롤러 유입률, 절대수익 자랑 노출 급증. 급증 시 게임이 사행/도박 프레임으로 새는 신호로 취급한다(앱스토어 심사 리스크·법적 가드레일).

> **마일스톤 게이트 분리 원칙:** 각 분기 마일스톤 게이트는 **①WAP 궤적**이라는 상위 게이트와 **②리텐션·바이럴 가설검증**(D7·K-factor 등, 추세·A/B로 검증)으로 **분리**한다. 리텐션 수치는 콘텐츠 물량 검증 전까지 게이트가 아니라 **검증 목표**로 관리한다(§16). 즉 "WAP가 오르는가"가 통과 조건이고, D7/K-factor는 같이 관찰하되 단독으로 릴리스를 막거나 통과시키지 않는다.

### 4.2.1 보조 KPI (계층별, 목표는 검증 대상 가설)

| 계층 | 지표 | 초기 목표(가설) | 근거 |
|---|---|---|---|

각 단계에 **들어가는 것 / 빠지는 것 / 빠지는 이유**를 명시한다.

### MVP (Q3 2026) — "혼자서도 돌아온다"
현재 코드는 트레이딩 코어만 있고 돌아올 이유가 없다. MVP 목표는 "혼자서도 3일은 돌아오게".

**IN:**
- (기존) 12종목 시뮬 마켓, 매수/매도, 포트폴리오, 거래내역, $100k — 유지.
- **Phase 0 회계 하드닝(버그성, UX 변화 0):** 실현손익 원장, 정수 센트 회계, 외부현금흐름(externalFlows) 분리, `schemaVersion`+마이그레이션 골격, UUID 거래 ID, buy/sell 원자화(리듀서), priceEngine seed 주입(결정론).

- **면책·연령게이트·개인정보:** 온보딩 미세 배너 + (결제/소셜 진입 직전) 전면 동의. 연령 수집은 **출생 연월**(일 미수집).

**OUT (이유):**
- 서버/인증/클라우드 동기화 — MVP 로컬 유지(라이선스 0, 규제 최소, 속도↑). 단 데이터 모델은 서버 이관 가능하게 설계.

## 6. 핵심 기능 상세

### 6.1 회계·트레이딩 코어 (Phase 0 — 최우선)
게임화·소셜·리더보드의 **하드 선행 과제**. 신규 UX 변화 없이 정확성만 확보한다.

| 결함 | 처방 |
|---|---|
| 실현손익 미집계(전량청산 시 손익 0) | `realizedPL` 원장 신설: 매도 시 `(fillPrice−avgCost)×qty − sellFees` 누적, 계좌·종목·체결 3층 기록 |
| 수익률 분모 유저 간 비교 불가 | (A)리더보드용 = 시작자본 $100k 고정 분모 + 외부흐름 차감, (B)실력 = TWR 이원화 |
| 외부현금흐름 오염 | 광고·미션 크레딧을 `externalFlows` 원장에 분리, cash 직접 가산 금지 |
| float 누적오차 | 정수 센트(USD=cent, crypto=sat) 저장, 표시 직전 변환, **round-half-even 단일 규칙** |
| buy/sell 레이스(더블탭) | 가드를 함수형 업데이터/리듀서 내부에서 원자 평가 + UUID+멱등키 |
| 시세 비결정론·휘발·과변동 | `price=f(seed,ticker,tickIndex)` 순수함수, 로그정규 GBM, 연환산 변동성 정규화, 소폭 드리프트, 영속/재계산 |

### 6.2 주문·체결 시스템
- **주문유형 축과 마찰 축 분리(중요 정합):** LIMIT/STOP/TAKE_PROFIT은 **쉬운 모드·무료**로 고정. 프리미엄/리얼모드 게이트는 **OCO/BRACKET/TRAILING·마찰(수수료·슬리피지)·대규모 대회개설**로 한정. (MON-01 매트릭스의 '지정가' 표기를 'OCO/고급 조건부 주문'으로 정정.)
- **도입 순서:** MARKET+LIMIT → STOP/TAKE_PROFIT → STOP_LIMIT/OCO/BRACKET/TRAILING.
- **대기주문 엔진:** 틱 갱신 직후 `evaluatePendingOrders`(순수함수, 결정론 우선순위). 틱 스냅 기반 판정(갭 체결은 현실 반영).
- **체결가:** 쉬운 모드 = mid 무마찰. 리얼 모드 = bid/ask 스프레드(변동성 연동) + 제곱근 시장충격 슬리피지. **대회 슬리피지는 본인 주문 notional만의 순수함수(개인 독립 유동성 풀, 공유 오더북 없음)** 로 결정론·감사 성립.
- **수수료·거래세:** 모드/시장별 파라미터. 매수수수료 자본화, 매도수수료·세 실현손익 차감. 롱 레버리지 **margin interest**도 숏 대차수수료와 대칭으로 절대시각 기준 일별 차감.
- **주문 확인 스텝:** 소액·시장가 1스텝, 고액·고급주문 2스텝(적응형 마찰).

### 6.3 포트폴리오·성과
- 총자산 = cash + Σ(mark×qty). **리더보드용 mark는 단일 정의(last)로 고정**(형평성). 미실현/총손익 = realized + unrealized.
- 위험조정 지표: MDD, 분산점수(1−HHI), 승률, 샤프유사(표본 부족 시 N/A). 자산곡선 스냅샷은 **절대 캘린더 시각** 기준(앱-오픈 의존 금지).

### 6.4 코퍼레이트 액션 (v2)
- 분할: `qty*=r`, costBasis 불변(avgCost는 `costBasis/qty` 파생) — **avgCost 스칼라 저장·조정 폐기**. 배당: **투자소득(endogenous)으로 총수익률 numerator에 포함**하되 광고·미션 크레딧만 external 차감(배당 이중 페널티 회피). 상폐: 마지막 가격 강제청산.

## 7. 사용자 경험·주요 화면·플로우

### 7.1 정보구조(IA) — 5탭 + 점진적 공개
```
[홈] [마켓] [챌린지] [랭킹] [내 자산]
 🏠   📈    🎯     🏆    💼
```
- **온보딩 직후:** 홈·마켓·자산 3탭만 노출. 챌린지·랭킹은 첫 미션 완료/서버 활성 이후 점진 공개(초보 위압감 최소화). 감춘 탭은 "준비 중" 빈상태를 명확히 설계.
- **챌린지 탭 역할:** 미션·챌린지·배지·시즌·수집 진행 **허브**. 선택적 팁/툴팁은 매매·포트폴리오 화면 **인라인(맥락형)**으로만 노출하고 별도 탭으로 격리하지 않는다.
- **기술 골격:** `useState` 탭 스위칭 → **Expo Router**(딥링크·웹 URL·뒤로가기 — 소셜 바이럴의 load-bearing 요소).

### 7.2 온보딩 — "30초 첫 매수 + 보장된 첫 승리"
1. 스플래시·가치제안 + 하단 미세 면책 + 앱스토어 연령등급 고지.
2. **게스트 즉시 진입** → 코치마크가 종목 지목, 잔고 충분한 수량 프리필.
3. 원탭 매수(튜토리얼 태그) → **첫 매수 컨페티 + '플레이 시작' 배지 획득 연출** + 다음 액션 넛지.
4. 홈 착지(오늘의 미션 1개 + 자산 스냅샷만; 위젯은 진도 연동 점진 노출).

**핵심 원칙(정합 확정):**
- **면책 시점(단일화):** 온보딩 시에는 **미세 배너만** 노출한다. **전면 면책 동의(체크박스+타임스탬프+버전 기록)** 는 **소셜·결제·대회 진입 직전에 트리거**한다. 순수 로컬 게스트(비수집·오프라인) 경험에는 미세 배너만 적용하고 전면 게이트를 걸지 않는다. "30초"는 이 순수 로컬·비수집 경험에만 적용된다.
- **연령 수집 정밀도(데이터 최소화):** 연령 입력은 **출생 연/월만** 받는다(일·정확한 생일은 미수집). 아동 식별정보 축적을 피하고 연령게이트(한14/EU16/미13, §13.3) 판정에 필요한 최소치만 보관한다.
- **첫 매수·첫 배지 획득을 명확히 축하**해 '시작의 쾌감'을 준다. 프리필 매수가 잔고 초과 시 자동 하향 조정(실패 불가 보장).
- (선택) 직후 "이 종목은 왜 이렇게 움직일까?" 같은 가벼운 팁 카드를 넛지로 제공할 수 있다(강제 아님, 스킵 가능).

### 7.3 첫 매매(정식) — 리서치→주문→확인
```
마켓 리스트 → [종목 상세: 차트·통계·내 보유·하단 고정 매수/매도] → [매매 시트]
 → 주문유형(시장가 기본 / 지정가·손절 고급) → 수량(±·MAX·금액역산)
 → [주문 확인: 예상가·수수료·잔고영향] → 비축하 토스트 → 상세 복귀
```

### 7.4 일일 복귀 — 홈
스트릭·오늘의 미션 2~3개·자산 스냅샷·(서버 후)리그 미니. **가드레일:** 데일리 미션 2~3개 상한, 푸시 1일 1~2회 상한, "연속 손실 시 잠깐 쉬기" 넛지(FCA '잦은 푸시' 안티패턴 회피).

### 7.5 빈/로딩/에러·디자인 시스템·접근성
- **로딩=스켈레톤**, **빈 상태 4요소**(아이콘+원인+다음행동+CTA), **에러=한국어 3원칙**(무슨일/왜/어떻게)로 현행 영문·무맥락('Not enough cash') 전면 교체.
- **등락 삼중 인코딩**(색+화살표▲▼+부호) → WCAG 1.4.1. 금액은 등폭(tabular-nums). `theme.ts`를 시맨틱 토큰으로 승격.
- **접근성 정합:** 라이브 리전은 **액션 결과(체결·에러)에만 assertive**, 2초 시세 갱신은 `aria-live=off`+온디맨드 announce(스크린리더 announcement 폭탄 방지). 배지 중첩(지연·모드·면책·출처) 시 시각 위계 붕괴 방지 위해 배지 예산 관리.
- **라이트/고대비 테마:** v1 다크 고정 가능하나 시맨틱 추상화 선반영, B2B 파일럿 전 라이트/고대비 최소 1종 준비.
- **현지화(로케일 우선순위 단일 참조):** 제품 UI 언어는 **①기기 로케일 → ②출시 시장 기준 → ③`en` 최종 폴백** 순으로 결정한다(미국 소프트런치 = 사실상 영어 우선, 기기 로케일이 지원 언어면 자동 선택). **"한국어 우선"은 내부 산출물 문서 작성 규칙일 뿐 제품 UI 언어가 아니며**, PRD·정책·유즈케이스 전 콘텐츠의 로케일 우선순위는 **본 §7.5를 단일 참조원(single source)** 으로 삼는다.

## 8. 게임화·소셜·경쟁 (핵심 가치)

> **이 섹션이 제품의 심장이다.** 트레이딩 코어는 게임판이고, 진짜 제품은 **레벨·배지·대회·리더보드·시즌으로 경쟁하고 자랑하고 수집하는 재미**다. 재미·경쟁·소셜·수집을 최우선 가치로 설계한다.

### 8.1 제1원칙 — 재미·경쟁·수집을 보상한다
스트릭·XP·배지·레벨·리그의 보상 대상은 **"게임을 즐기고, 경쟁하고, 진행하고, 수집하는 것"** 이다. 매일 돌아올 이유, 친구를 이길 이유, 컬렉션을 채울 이유를 겹겹이 쌓는다.

- **단, 법적 가드레일은 지킨다(재미 게임이어도):** ①실제 현금 상금 대회 없음, ②"모의투자 시뮬레이션" 면책 상시, ③개인 맞춤 매수 추천 없음, ④실데이터 라이선스 확인. 또한 앱스토어 **사행성(도박) 심사 리스크**를 피하려고 순위는 절대수익 단일이 아니라 복합 스코어를 병용한다(§8.4). 이는 재미를 죽이는 규칙이 아니라 **스토어에서 살아남기 위한 최소 방어선**일 뿐이다.

### 8.2 3종 화폐 분리
| 화폐 | 획득 | 용도 | 하드 가드레일 |
|---|---|---|---|
| **XP** | 거래·대회·미션·배지 등 **게임 플레이** | 레벨·리그 산정 | — |
| **가상현금 $** | 초기 $100k, 미션/대회 보너스 | 매매 원금 | 손실 무게감 유지(게임 긴장감) |
| **코인/젬** | 리워드 광고·IAP·업적 | 코스메틱·편의 | **코인으로 가상현금 구매 불가**(현금상금·사행성 우회 방지) |

### 8.3 스트릭·배지·미션
- **스트릭:** "매일 돌아오기"를 보상 — 로그인 + 아무 게임 액션(거래·미션·대회 확인) 1개면 유지. 프리즈 필수(무료 주1회+미션 보상; **직접 유료판매 배제** — 손실회피 수익화 다크패턴 회피). 위켄드 앰네스티, 장기 미접속자 압박 알림 중단(웰빙 가드).
- **배지:** 수집·탐험·경쟁·마일스톤·시즌 등 **폭넓은 컬렉션**으로 "채우는 재미"를 극대화. 단 노골적 도박 조장 배지(몰빵 100%·하루 20거래 등)는 **카탈로그에서 원천 배제**(사행성 심사 방어).
- **축하 연출:** 첫 매수·수익·레벨업·배지·대회 승리·시즌 완주 등 **성취를 시원하게 축하**(컨페티·연출)해 쾌감을 준다. 단 "손실 후 즉시 재매수 유도" 같은 loss-chasing 유발 연출은 배제(웰빙·심사 가드).
- **데일리 미션 다양성:** 미션을 "많이 사기"류로 단조롭게 반복하면 지루하고 사행 프레임도 커짐 → "새 종목 탐험·대회 참여·컬렉션 채우기·챌린지 클리어" 등 **다양한 재미 축**으로 구성.

### 8.4 리더보드·리그
- **절대수익률 단일 순위 금지.** 복합 스코어(수익률×리스크조정계수 + 분산·규칙준수, 손실 구간엔 리스크계수 곱 미적용해 부호역전 방지). 순위 옆 MDD·최대 단일종목 비중 병기.
- **리그:** 약 30명 소규모 풀 + 주간 승급/강등. 초보/숙련·쉬운/리얼·(향후)소액모드 분리.
- **지표 게이밍 주의(2차효과):** 인위적 저변동 후 막판 몰빵, HHI 채우기용 무의미 분산 → 기간 일관성·규칙준수율 가중으로 게이밍 저항성 확보.
- **시뮬 통계 유효성:** 평균회귀 시계열의 음의 자기상관으로 샤프 과대평가 위험 → 스냅샷 절대시각 고정, 최소 표본·장기 구간에서만 샤프 계산.

### 8.5 대회·소셜
- **비공개 그룹 대회(최우선 소셜·K-factor 엔진):** 방장이 파라미터화, 서버 시드 결정론 시세. Free도 소규모(≤3명) 1개 개설 허용(바이럴 앵커).
- **유료 참가비+현금 상금 대회 최소 1년 배제**(복권 3요소·랜덤워크 실력논거 취약).
- **바이럴 앵커:** '수익률 자랑'과 **'초대 링크(그룹 대회 참가)'** 를 함께 민다. 공유 카드는 배지·레벨·리그 티어·대회 성적 중심, 하단 면책 강제(가드레일 ②). K-factor 목표는 초대 루프 기준으로 산정.
- **활동 피드:** 배지·레벨업·승급·스트릭·챌린지 완주·대회 승리 노출. 절대수익 자랑은 노출 절제(herding·사행 프레임 완화). 자동 카피트레이딩 금지, 인사이트는 **일반·비개인화**만 허용(개인 맞춤 매수 추천 금지 — 가드레일 ③).
- **신고·차단·모더레이션(소셜 v1 하드 선행 게이트):** 모든 UGC·소셜 노출 표면(공유 카드, 활동 피드, 그룹 대회, 친구·닉네임)에 **신고 → 모더레이션 큐 → 차단/제재** 파이프라인을 표준 기능으로 편입한다. 이는 소셜 v1의 **하드 선행 요건**(LEG-06 정합)으로, 파이프라인 미비 시 해당 소셜 표면을 공개하지 않는다. 실존 자산 매수 선동·시세조종성 게시는 금지(크립토 실데이터 후 모더레이션 강화, §13.4). 상세 흐름은 use-cases.md(UC-B29~B31) 참조.

### 8.6 서버 의존성 & 임계 선행작업
소셜/리그는 서버 없이는 구현 불가(로드맵 종속성). 임계 선행: **다계좌 리팩터**, **서버 사이드 이벤트 검증 로그**(치팅 방지), **서버 UTC 시계**(스트릭·시즌·마감 판정). 하이롤러 리그 등 규칙 변경은 **시즌 종료 후 적용**(시즌 중 규칙 불변).

## 9. 선택적 팁·툴팁

> 이 제품은 교육 앱이 아니라 **재미 게임**이다. 학습은 **핵심이 아니며**, 원하는 유저를 위한 **선택적·비강제 팁** 수준으로만 존재한다. 어떤 팁도 온보딩·진행을 막지 않고 언제든 스킵할 수 있다. 리텐션은 팁이 아니라 게임 콘텐츠(§8)로 만든다.

### 9.1 팁의 범위와 금지선
- **가벼운 팁(선택):** "분산이 뭔가요", "손절 주문은 이렇게", "이 종목은 왜 이렇게 움직일까" 같은 30초 툴팁·용어 설명을 매매/포트폴리오 화면 **인라인(맥락형)** 으로만 제공. 별도 학습 탭·커리큘럼·수료 트랙·학습성과 측정은 없다.
- **법적 금지선(가드레일 ③):** 개인 맞춤 매수 추천·"지금 X 사라"·종목 예측·목표주가·개인화 실시간 시그널은 **어떤 형태로도 제공하지 않는다.** 팁은 일반론·비개인화만. 종목명은 예시로만, "일반 정보·비자문" 라벨 고정.

### 9.2 전달 방식
- **맥락형 툴팁:** 매매/포트폴리오 용어에 점선 밑줄 → 30초 카드 바텀시트(별도 탭 격리 안 함, 스킵 가능).
- **선택적 팁 카드:** 챌린지 탭 안의 가벼운 컬렉션(24장 내외). 읽으면 소량 코스메틱 보상이 있을 수 있으나 **게임 진행에 필수가 아니다.**
- **시나리오(재미 콘텐츠):** `priceEngine` 래핑 `ScenarioEngine`으로 급락장·변동성 폭발·섹터 로테이션을 **재미있는 도전 모드**로 재생("급락장에서 살아남기" 챌린지 등). "가상 시나리오" 라벨.

### 9.3 콘텐츠 운영(경량)
- **Content-as-Data:** 팁 카드·용어·시나리오·튜토리얼을 JSON/MDX로 관리. 번들 기본값 + 원격 오버라이드(앱 재심사 없이 갱신, 오프라인 폴백). v1은 Git 기반(PR=검수 게이트).
- **법무 검수 게이트(머지 필수):** 개별 매매 지시·예측·수익보장·실시간 시그널 금지, '일반 정보·비자문·모의' 라벨 상시(가드레일 ②③). AI 도우미는 사전정의 답변 우선 + 자유생성 범위 제한 + 오정보(hallucination) 면책.
- **콘텐츠 물량은 게임 콘텐츠로 해결:** 리텐션은 팁이 아니라 **미션·챌린지·시즌·대회·수집** 로테이션으로 확보한다(§8). 무거운 학습 인력(교수설계자·SME 전담)·문항은행·SRS·학습성과 대시보드는 **범위에서 제외**한다.

## 10. 기술 아키텍처·스택 결정과 근거

### 10.1 스택 결정

| 영역 | 결정 | 근거 |
|---|---|---|
| 런타임 | Expo 유지, Expo Go→**dev-client** | 광고/IAP/MMKV 네이티브 모듈 필요 시점에 선제 전환 |
| 내비게이션 | **Expo Router** | 그룹/대회 초대 딥링크·웹 동일 라우트 = K-factor load-bearing |
| 상태관리 | **Zustand(클라)+TanStack Query(서버)** | 2초 tick 전 consumer 재렌더 폭풍 해소(selector 구독) |
| 로컬 저장 | **MMKV(핫 KV)+expo-sqlite(원장·시계열)**, AsyncStorage 폐기 | 무한 성장 blob·부분쓰기·경쟁 제거. 웹은 StorageProvider 추상화(IndexedDB 폴백) |
| 차트 | **victory-native XL(Skia)** + Skia 스파크라인 | New Arch·웹 호환 |
| 도메인 | **순수 TS `domain/` 레이어(RN 의존 0)** | 클라/서버 동일 실행 → 대회 공정성·치팅방지·"실력 콘테스트" 근거 |
| 백엔드-데이터평면 | **Supabase(Postgres+Auth+RLS+Realtime+Edge Functions)** | Auth/게스트→가입, **RLS 데이터 경계**, **이벤트소싱 원장**(append-only), 리더보드 **집계 SQL·머티리얼라이즈드 뷰**, 정적 조회. 실시간 시세 스트림은 **여기서 하지 않는다**(§10.3) |
| 백엔드-실시간평면 | **상시구동 워커(long-lived) + Redis(또는 대체)** — Fly.io/Railway/Render 상주 컨테이너 또는 Cloudflare Durable Objects | 거래소 WS 업스트림 흡수, **시세/팬아웃 워커**, hot-cache·요청병합, 대회 틱 스트림, 서킷브레이커 상태, 이상치 2-틱 버퍼(§10.3). 솔로는 미경유, **소셜/대회를 켤 때만 진입** |
| 수익화 | **RevenueCat(구독/IAP 크로스플랫폼 검증)+AdMob** | 서버 영수증 검증·엔타이틀먼트 |
| 배포 | **EAS Build 3프로파일+Update(runtimeVersion=fingerprint)+Submit+CI 게이트** | OTA는 버그픽스·콘텐츠 한정(심사 우회 배포 금지) |

### 10.2 아키텍처 원칙
- **`domain/` 순수 TS 레이어**에 주문엔진·시뮬레이터·P&L·샤프/MDD·코퍼레이트액션·`Money` 값객체(round-half-even 단일 규칙) 분리 → 클라·서버 동일 구현 공유(±1센트 발산 방지).
- **`MarketDataProvider` 인터페이스**(`getQuote/subscribe/getCandles`)를 지금 도입, `SimulationProvider` 첫 구현체. API 키는 **반드시 백엔드 프록시 경유**(키 은닉+캐시+요청병합+서킷브레이커→시뮬 폴백).
- **이벤트소싱 거래원장**(UUID+멱등키, append-only) + 아웃박스 동기화. 솔로=로컬 우선(오프라인 완전 작동), 대회=서버 권위.
- **다계좌·엔타이틀먼트 게이팅**을 v1 진입 시 선추상화(현행 단일 로컬계정 리팩터).
- **확장성:** FlashList v2 + 뷰포트 기반 시세 구독(가시/보유 종목만 고빈도), SQLite 마이그레이션 러너.

### 10.3 백엔드 아키텍처 현실 정합 — 상시 워커 vs 관리형, 결정론 재계산 비용

> **왜 이 절이 필요한가.** §10.1은 백엔드를 "Supabase(Postgres+Auth+RLS+Realtime+Edge Functions)"로 단일 확정했었고, 시세 결정론의 부수효과로 "과거 임의 시각 가격을 **O(1) 재계산**"할 수 있다는 가정이 있었다. 두 진술 모두 정책이 요구하는 실시간 대회·리더보드와 채택한 시뮬 엔진(§11.2)의 실제 시간복잡도 앞에서 무너진다. 이 절은 두 결함을 규칙·수치로 못박아, 실시간 대회·비용모델·동기화 설계가 잘못된 전제 위에 쌓이지 않게 한다. 상세 근거는 [planning-review.md](./planning-review.md).

#### 10.3.1 결함 A — 관리형 stateless로는 정책이 요구하는 상시 스트림을 호스팅할 수 없다

정책(MKT-07)은 "서버가 **업스트림 1개 구독 → N 유저 브로드캐스트**"(WS 팬아웃), "인기 종목 **hot-cache TTL 1~5초**", "대회 시작 시 **동시 구독 폭주 큐잉**"을 요구한다. 이들은 전부 **프로세스가 틱과 틱 사이에 살아 있어야(stateful, long-lived)** 성립한다. 반면 Supabase Edge Functions는 요청당 부팅·수십 초 실행상한의 **stateless FaaS**이며, Supabase Realtime은 "Postgres 행 변경 브로드캐스트"이지 "외부 WS를 흡수해 재분배하는 팬아웃 게이트웨이"가 아니다.

| 정책이 요구하는 것 | 필요한 실행형태 | Supabase Edge/Realtime 단독 가부 |
|---|---|---|
| 업스트림(거래소 WS) 1구독 → N유저 팬아웃 (MKT-07) | 지속 연결을 쥔 **상시 프로세스** | ✗ (Edge는 요청 종료 시 연결 소멸) |
| hot-cache TTL 1~5초, 요청병합(fan-in) (MKT-07) | 인메모리/Redis **공유 상태** | ✗ (Edge 인스턴스 간 상태 공유 없음) |
| 대회 틱 스트림 서버 권위 팬아웃 (MKT-01/12) | 틱 루프를 도는 **워커** | ✗ |
| 서킷브레이커 half-open 히스테리시스 (MKT-07) | 프로세스 수명 넘는 **카운터** | △ (DB 왕복 시 지연·비용) |
| 이상치 2-틱 확인 로직 (MKT-08) | 직전 틱을 **기억하는 상태** | ✗ |

**결정(§10.1 백엔드 행 반영):** 백엔드를 **역할별 2계층**(데이터평면 / 실시간평면)으로 분리 명시했다(§10.1 표). 하나의 관리형 서비스에 실시간 스트림을 억지로 태우지 않는다. 담당 범위는 §10.1 표의 두 행과 같다.

- **Redis 대체 허용:** 초기 규모에선 단일 워커 프로세스 인메모리 맵으로 시작 가능(수평확장 0). Redis는 **워커 2대 이상으로 수평확장하거나 대회 동접이 게이트(10.3.3)를 넘는 시점**에만 도입 — 관리형 Upstash(요청과금) 또는 워커 사이드카.
- **비용 하한이 0이 아님을 재무모델에 반영:** 상시 워커는 최소 1대가 24/7 과금(대략 $5~25/월/워커 + Redis)이다. 손익분기 역산·§12 수익화는 이 **인프라 고정비 하한**을 전제로 다시 계산한다(§12.6). 현재 "솔로는 백엔드 0"은 여전히 참이나, **소셜/대회를 켜는 순간 0이 아니다**.
- **솔로 경로는 무변경:** 솔로(연습) 모드는 클라이언트 결정론(§11.1)으로 백엔드 0 유지. 실시간평면은 **대회/리그/리더보드/실데이터**를 켤 때만 진입한다.

#### 10.3.2 결함 B — "O(1) 재계산"은 신화다. 채택 엔진은 O(tickIndex)

"과거 임의 시각 가격 O(1) 재계산"은 결정론의 공짜 이득처럼 인용됐으나, 이는 **가격이 tickIndex의 무상태 순수함수**일 때만 참이다. 그러나 §11.2가 채택한 엔진은 **경로의존(path-dependent)** 이다.

- **GARCH-lite 변동성:** `σ_t² = ω + α·r_{t-1}² + β·σ_{t-1}²` — σ_t가 σ_{t-1}에 재귀 의존.
- **평균회귀/EMA 추세 진동:** EMA 자체가 전체 과거의 지수가중합.
- **상관 팩터 상태:** 공통 시장팩터·섹터팩터 레벨이 스텝마다 누적.

따라서 `price(instrument, T)`를 얻으려면 시작점부터 T까지 **모든 틱을 순차 전개**해야 한다 → 실제 복잡도는 **O(T − T₀)**, 즉 O(tickIndex). "O(1)"이 아니다.

**이 신화가 부수는 것 — 구체 수치(TICK_MS=2000, 즉 0.5틱/초):**

| 시나리오 | 재계산 틱 수(앵커 없음) | 종목 M=50 시 총 반복 |
|---|---|---|
| 앱 6시간 방치 후 복귀 | 6h×1800 = **10,800** | 540,000 |
| 주말(48h) 방치 크립토 24/7 | 48h×1800 = **86,400** | 4,320,000 |
| 장기 미접속 3개월 크립토 | 90d×43200 = **≈3,888,000** | **≈194,000,000** |

한 화면을 그리려 종목당 수백만 번의 GBM+GARCH 반복 → 콜드스타트 프리즈·배터리 소진·대회 서버 CPU 폭발. "결정론이면 저장 대신 재계산" 문장은 이 비용 없이는 성립하지 않는다.

**결정 — 앵커 체크포인트를 엔진 불변식으로 의무화:** 결정론 엔진은 **주기적 앵커 스냅샷**을 남기고, 모든 재계산을 **최근 앵커 ~ 현재**로 **클램프**한다.

- **앵커 스냅샷 페이로드(종목별):** `{ tickIndex, price(정수 최소단위), σ_t², factorState(공통·섹터 레벨), rngState(PRNG 내부상태), seedVersion, epochAnchorMs }`. 이 7요소가 있으면 어떤 시각도 앵커에서 이어 전개 가능.
- **앵커 케이던스:** `ANCHOR_INTERVAL_TICKS = 450`(2초틱 기준 15분) **또는 세션 경계 중 먼저 오는 것**. 즉 앵커는 `max(직전 세션개장, 15분마다)`.
- **재계산 상한 보장:** 어떤 콜드스타트도 **최대 450틱 × M종목**만 전개 → 위 표의 3개월 케이스가 종목당 4M회에서 **450회**로 축소(약 8,600배 감소). 실질 O(1) 상한을 규칙으로 만든다.
- **자산군별 앵커 규칙(세션모델과 결합):**
  - **EQUITY/ETF:** 마감 중 틱 정지 → σ·팩터상태가 동결되므로 **세션 개장이 자연 앵커**. 재계산은 당일 개장~현재로 자동 클램프.
  - **CRYPTO 24/7:** 세션 경계가 없어 자연 앵커가 **영원히 안 옴** → **15분 강제 앵커를 필수**로 명시(강제 없으면 위 3개월=4M 반복 케이스가 그대로 현실화). 이 한 줄이 이 절의 가장 load-bearing한 규칙이다.
- **저장 위치(신뢰 경계별):**
  - 솔로/클라: 최근 앵커 1개를 MMKV에 로컬 저장(§10.1). 콜드스타트 시 `nearestAnchor ≤ now`에서 전개.
  - 대회/서버: 앵커를 Postgres에 **append-only로 영속**(감사·분쟁 재생성 요건 충족). 분쟁 리플레이 = `앵커 + seed`로 결정론 재생.
- **감사 정합:** 앵커는 seedVersion을 담으므로 §11.1의 "seedVersion pin"과 충돌 없이 버전 경계에서 새 앵커 라인 시작.

#### 10.3.3 결함 C(재무·용량) — 대회 부하는 산정 대상이며 로드맵 게이트다

서버 도입을 "언제"만 묻고 "**어떤 형태**"(관리형만 vs 상시 워커 포함)와 "**어느 부하까지**"를 안 물으면, 소셜을 켜는 순간 미검증 용량으로 진입한다. 대회 부하를 **파라미터화된 산정식**으로 고정하고 **로드맵 게이트**로 건다.

**부하 산정식(동시 진행 대회 기준):**

```
활성종목 가격연산/초   = Σ대회( 유니버스종목수 M ) × (1 / TICK_초)      # 팬아웃 덕에 참가자 수와 무관
WS 팬아웃 메시지/초     = Σ대회( 동접참가자 × 1인구독종목 ) × (1 / TICK_초)
리더보드 재계산/초      = Σ대회( 동접참가자 × 평균보유종목 ) × 마크투마켓빈도
동시 WS 연결 수        = Σ대회( 동접참가자 )
```

**기준 시나리오(TICK_초=2, 대회당 참가 30명·동접 50%·유니버스 M=50·1인 구독 20종목·보유 8):**

| 동시 대회 N | 가격연산/초 | 팬아웃 msg/초 | 동시 WS 연결 | 판정 |
|---|---|---|---|---|
| 10 | 250 | 1,500 | 150 | 단일 워커 인메모리로 충분(Redis 불요) |
| 100 | 2,500 | 15,000 | 1,500 | 워커 1~2대·Redis 도입 라인 |
| 500 | 12,500 | 75,000 | 7,500 | 워커 수평확장·Redis 필수·큐잉(MKT-07) |

- **핵심 통찰:** 팬아웃 구조(MKT-07) 덕에 **가격연산은 참가자 수와 무관**(종목당 틱 1회)하지만, **네트워크 egress와 WS 연결 수는 동접에 선형**이다. 병목은 CPU가 아니라 **연결·egress**다 → 워커 스펙은 vCPU가 아니라 **동시연결·대역폭 기준**으로 사이징한다.
- **대회 시작 스파이크(MKT-07 "동시 구독 폭주"):** N개 대회가 정각 동시 시작하면 순간 구독요청이 위 정상치의 수십 배 → **백엔드 큐잉+우선순위(보유>가시>기타)** 를 게이트 통과 조건으로 명시.

**로드맵 게이트(§17 소셜/대회 마일스톤 진입 전 필수 통과):**

| 게이트 | 통과 기준 |
|---|---|
| G1 부하 재현 | 위 "N=100" 시나리오를 합성부하로 재현, p99 틱 지연 < 250ms |
| G2 팬아웃 검증 | 업스트림 1구독 → 30유저 브로드캐스트 실측(심볼 한도 내), 중복 업스트림 0 |
| G3 앵커 콜드스타트 | 크립토 48h 방치 계좌 콜드스타트 재계산 < 100ms(앵커 클램프 실측) |
| G4 비용 하한 | N=100 시 워커+Redis 월비용 산정치를 §12.6 손익분기에 반영 |
| G5 큐잉 | N개 대회 동시 시작 스파이크에서 드랍 0·우선순위 순서 보장 |

> **관련 정책 반영:** policies.md의 MKT-01(재계산 복잡도·앵커 체크포인트), MKT-07(상시 워커+Redis 전제·큐잉)에 본 절의 규칙을 규칙/사이드이펙트로 반영한다. 상세 근거는 [planning-review.md](./planning-review.md).

## 11. 데이터·시세 전략

### 11.1 시세 권위·결정론
- 기기-로컬 `Math.random()` → **`price=f(seed,ticker,tickIndex)` 서버/NTP 절대시각 순수함수.** 솔로=클라 결정론(비보상), 대회·리그=**서버 은닉 시드**(미래가 예측 치팅 차단). 로컬 결정론 시세로 만든 프로필 실력지표·공유 카드는 순위 미연동·신뢰 경계 명시.
- **결정론 재계산은 O(1)이 아니라 O(tickIndex)다(§10.3.2).** 채택한 GARCH-lite·평균회귀·팩터 상태가 순차의존이므로, 시세를 '저장 안 하고 임의 시각 O(1) 재계산'한다는 가정은 폐기한다. 대신 **주기적 앵커 체크포인트(15분/세션경계, `{tickIndex·price·σ_t²·factorState·rngState·seedVersion·epochAnchorMs}` 7요소 스냅샷, §10.3.2)** 를 남기고 재계산을 **'최근 앵커~현재'로 클램프**한다(재계산 상한 ≤ ANCHOR_INTERVAL 틱). **크립토 24/7은 세션 앵커가 없어 15분 강제 앵커가 필수**다. 세션 경계에 틱이 멈추는 EQUITY/ETF는 세션 개장이 자연 앵커가 된다.

### 11.2 현실성 캘리브레이션
- 현행 `shock=U(-1,1)*(vol/100)`은 시간축 오정규화. → 로그정규 GBM, 연환산 σ(AAPL 0.30 등)·소폭 드리프트(+7%/년, 상한). **주의:** 데이터 엔지니어의 "연 595%·20배 과변동"은 평균회귀 제거 시 무제약 확산치이며, 실제 현행 엔진은 κ=0.02 평균회귀로 밴드에 갇힘(저변동 종목은 오히려 실현 변동이 작음) → GBM 전환 시 annualVol을 **실현 목표로 캘리브레이션** 필수(내부 모순 회피).

### 11.3 소스 티어링 로드맵
| 단계 | 미국주식 | 크립토 | 한국/기타 | 비용 |
|---|---|---|---|---|
| P0(MVP) | 시뮬(결정론) | 시뮬 | 시뮬 | $0, 규제 최소 |
| P1(베타) | Finnhub 무료(내부 테스트만) | 거래소 WS(**약관 개별 법무검토 후**) | 시뮬/지연 | $0(재배포 미허용 주의) |
| P2(정식) | Polygon/TwelveData 재배포 유료 | 거래소($0) | KRX 지연/일봉 | $30~200/월 |
| P3(스케일) | Polygon 실시간 재배포 | 거래소 WS | KRX 실시간(별도 견적) | $200+/월 |

> **'비용' 열 각주:** 위 데이터 라이선스 비용은 **호출량 과금 + 팬인/캐싱 전제(시나리오 A)** 의 값이다. per-subscriber 계약이면 MAU에 비례해 폭증하며, 실시간 전송(WS egress) 인프라는 별도로 경쟁모드 동접에 비례한다. 손익분기·스케일 모델은 **§12.6** 참조.

- **재배포 게이트:** 무료 티어 실시간의 공개 배포는 대부분 약관 위반 → 정식 출시는 재배포 명시 허용 플랜만. 크립토 "재배포 완화" 가정은 **계약 조항 확인 전까지 폐기.** 실데이터 대회는 시드 재생성 불가하므로 **녹화된 서버 권위 틱 로그(해시체인 감사)** 를 진실원본으로.
- **오인 방지:** `Quote.source/delayedMs` 배지("실시간/△분 지연/시뮬"). 지연 데이터 실시간 오인 표시 금지. 단 "15분 지연" 배지가 몰입을 갉을 수 있음(2차효과) → 노출 절제.
- **시장 세션:** MVP는 24/7 시뮬 기본 옵션, `SessionSpec` 필드 예약. 24/7에서도 UTC '거래일' 경계로 dayOpen·DAY TIF 정의.

### 12.4 유닛 이코노믹스 (보수적)

> **재무 SSOT:** ARPU·LTV·손익분기(BEP)·B2B ARR 등 **모든 재무 수치의 단일 소스는 [monetization-model.md](./monetization-model.md)** 다. 본 절은 그 값을 **전재(인용)** 만 하고 자체 값을 두지 않는다. 모든 수치는 별도 표기 없으면 **net**(스토어 수수료 SBP 15% + 환불·미수 ~3% 반영 후 ≈ gross×0.825)이며, `gross`는 유저 청구액으로 라벨을 붙인다.

- **광고 ARPU 보수 재추정:** 종전 광고 ARPU $0.10(gross 전제·가드레일 미반영)은 폐기. 리워드 가드레일·비개인화 폴백·미성년/학생 카테고리 차단 반영 시 **광고 net ARPU = 보수 $0.040 / 기본 $0.050 / 낙관 $0.070**.
- **구독 ARPU 정정:** 종전 "$0.06"은 라벨 부재·산술 오기였다. 정정값 = **구독 net $0.04**(= gross $0.05 = 인스톨→유료 2% × 결제자 월 net ~$2.5).
- **블렌디드 net ARPU 3시나리오(monetization-model.md §2.3 전재, net/gross 라벨 병기):**

  | 시나리오 | 구독(net) | 광고(net) | IAP(net) | **블렌디드 net ARPU** | $30k net MRR 도달 MAU(매출 마일스톤, ≠손익분기) |
  |---|---|---|---|---|---|
  | 보수 | $0.040 | $0.040 | $0.020 | **$0.100** | ~300k |
  | 기본 | $0.041 | $0.050 | $0.025 | **$0.116** | ~259k |
  | 낙관 | $0.045 | $0.070 | $0.030 | **$0.145** | ~207k |

- **라벨 정정(중요):** 위 마지막 열은 **"$30k net MRR에 도달하는 MAU"(매출 마일스톤)** 이지 **손익분기(BEP)가 아니다.** 종전 표는 이를 'BEP'로 오라벨(그리고 낙관 $0.18/167k 등 팬텀·라벨 밀림)했다. **실제 손익분기(monetization-model.md §4):** 인프라 단독 BEP는 **~5~7k MAU**(초기 생존 용이), 팀 인건비 포함 BEP는 **~190k(낙관)/242k(기본)/285k(보수) MAU**(B2B ARR 반영 시 ~24k MAU 하락). 종전 "$30k MRR에 150k MAU"는 gross $0.19 기준값으로, net 기준으로는 ~259k(기본)다.
- **함의:** **규모(오가닉·바이럴·B2B)가 수익화보다 선행 병목**이다(인스톨당 net LTV $0.63 → 유료 UA는 현 이코노믹스에서 성립 불가). **구독·B2B 중심 시나리오를 기본안으로** 재모델링한다(광고는 보조 업사이드, §16 O2). MRR 목표를 광고 낙관에 의존시키지 않는다.
- **앱스토어 수수료:** 초기 SBP 15%로 모델링, 연매출 $100만 돌파 시 Apple 30% 구간 시나리오. 웹 결제(~3%)는 반스티어링·EU DMA 개별 검증 후에만.
- **데이터·실시간 인프라 = 고정비가 아니라 리스크 축:** 데이터·실시간 인프라는 '관리 가능한 고정비'가 아니라 **과금모델(호출량 vs per-user)·심볼 유니버스·동접 스케일에 따라 변동하는 리스크 축**이다. 호출량 과금 + 캐싱 하에서는 데이터가 월 $199 내외로 평탄(MAU 무관)하지만, **per-subscriber(유저당) 실데이터 계약이면 MAU에 비례해 폭증**(150k MAU × $1.5 = 월 $225k → 매출 8배 초과, 영구 적자)한다. per-user 데이터 계약은 계약 게이트(§12.6 IC-6)에서 원천 차단한다. 손익분기 유저수 표와 스케일 모델은 **§12.6** 참조. 진짜 고정비 리스크는 UA와 **경쟁모드 실시간 전송(WS egress) 인프라**다.

### 12.5 B2B
- 앱 내 IAP가 아닌 **기업 계약·인보이스 연간 시트 라이선스**(스토어 수수료·자문 규제 우회). 학생 시트는 광고·결제·공개 소셜 코드레벨 비활성. 파일럿도 **DPA/학생데이터 처리계약 선체결 후** 시작.

### 12.6 인프라·실시간 데이터 비용 스케일 모델 (동접·데이터 과금·손익분기)

> **정합:** §12.4의 "데이터는 월 $30~200 관리 가능"은 **무조건이 아니라 과금모델·캐싱·스케일에 조건부**다. 호출량 과금 + 백엔드 팬인/캐싱 + 뷰포트 티어링이 모두 성립할 때만 참이며, per-subscriber(유저당) 계약이면 MAU에 비례해 폭증한다(정책 MKT-06 사이드이펙트와 정합). 이 절이 §12.4의 정량 근거이자 §11.3 표 '비용' 열의 참조원이다.

#### 12.6.1 제1원칙 — 비용은 '모드'로 분기한다 (솔로=0 / 경쟁=서버권위)

인프라 비용의 90%+ 유저(캐주얼)는 **한계비용 0**이어야 한다. 이를 아키텍처 규칙으로 못박는다.

| 모드 | 시세 권위 | 백엔드 부하 | per-user 한계비용 | 근거 정책 |
|---|---|---|---|---|
| **솔로/연습** (기본, 90%+) | 클라이언트 결정론 `f(seed,ticker,tickIndex)` | **0** (백엔드 미경유) | **$0** | MKT-01, 순위 미연동 |
| **주간 리그(비현금)** | 서버 권위 스냅샷(집계) | 스냅샷 평가만(WS 없음 가능) | 낮음(캐시 CPU) | GRF-02/04 |
| **실시간 대회** | 서버 권위 WS 팬아웃 | WS egress + 평가 | **동접에 비례** | MKT-01/07, GRF-11 |

- **규칙 IC-1(모드 게이팅):** 서버 권위 실시간 스트림은 **순위·보상이 걸린 경쟁 컨텍스트에만** 켠다. 솔로 홈·워치리스트·종목상세는 클라 결정론(시뮬) 또는 지연 캐시로 서비스하며 WS를 붙이지 않는다. → 실시간 인프라 청구가 되는 유저 = 전체 MAU가 아니라 **동시에 대회에 접속한 부분집합**뿐.

#### 12.6.2 실시간 전송(WS egress) 비용 — 동접별 산식

경쟁모드 실시간 전송비의 지배항은 **egress 대역폭**이다. 산식:

```
egress_bytes/s = U × V_eff × B × R_eff
  U      = 동시접속(경쟁모드) 유저 수
  V_eff  = 유저당 실제 푸시 심볼 수 (뷰포트 티어링 후)
  B      = 심볼 델타 페이로드 (~50B, 프레이밍 포함)
  R_eff  = 심볼당 유효 푸시 주기(Hz) — N초 집계로 상한
월 egress(GB) = egress_bytes/s × 2.59e6(s/월) / 1e9
월 egress비($) ≈ 월 egress(GB) × $0.09/GB (표준 클라우드 egress, 추정)
```

**캡 적용값(권고 운영점):** 뷰포트 티어링(가시 ~6종목 @0.5Hz + 보유 ~4종목 @0.2Hz) + N초 집계(N=2s) → **유저당 ≈ 200 B/s**.

| 동접(경쟁) | egress(캡 적용) | 월 egress | egress비/월 | 팬아웃·평가 컴퓨트 | 캐시·베이스라인 | **인프라 합계/월(추정)** |
|---|---|---|---|---|---|---|
| **5k** | 1.0 MB/s | 2.6 TB | ~$230 | ~$150 | ~$100 | **~$480** |
| **20k** | 4.0 MB/s | 10.4 TB | ~$930 | ~$400 | ~$200 | **~$1,530** |
| **50k** | 10 MB/s | 25.9 TB | ~$2,330 | ~$900 | ~$400 | **~$3,630** |

**캡 미적용(나이브: 보유+워치 ~30종목을 매 업스트림 틱 1Hz 그대로 푸시)** 시 유저당 ~1,500 B/s(7.5×) → **50k 동접 egress만 월 ~$17,500**. 즉 아래 캡들이 비용을 **7~8배** 절감하는 게 아니라 **폭증을 막는 상한선**이다.

- **캡 IC-2(N초 집계):** 서버는 업스트림 틱 속도와 무관하게 **심볼당 최대 1개 병합 스냅샷/ N초**만 팬아웃한다(N=2s 기본, 대회 막판 1s로 가변). → egress를 업스트림 틱 레이트에서 **분리**.
- **캡 IC-3(뷰포트 티어링):** 가시 종목 1~2s / 보유 5s / 나머지 30s·온디맨드, 뷰포트 밖은 고빈도 구독 해제(FlashList 가시성 연동, MKT-07). → egress를 심볼 유니버스 크기에서 **분리**.
- **캡 IC-4(관리형 실시간 회피 기준):** Ably/Pusher류 **동시연결·메시지 종량** 관리형 서비스는 5k 동접부터 자체호스팅 WS(Fargate/VPS + Redis pub/sub) 대비 급격히 비싸질 수 있음 → 5k 동접 초과 예상 시 자체호스팅으로 전환하고, 관리형은 5k 이하 MVP·대회 파일럿에만 사용.

#### 12.6.3 리더보드 스냅샷 평가 빈도 (컴퓨트 상한)

평가 컴퓨트는 **마크가격이 hot-cache에 있으면 CPU-only로 저렴**하다. 병목은 egress이지 평가가 아니다 — 단, 평가 빈도를 컨텍스트별로 티어링해 낭비를 막는다.

| 컨텍스트 | 평가 주기 | 방식 | 비고 |
|---|---|---|---|
| 실시간 대회(진행 중, 막판 1시간) | **10s** + 체결 이벤트 시 | 서버 재계산 | 순위 확정은 서버권위가만 사용(GRF-02) |
| 실시간 대회(그 외 구간) | 60s + 체결 이벤트 시 | 서버 재계산 | |
| 주간 리그 순위표 | **5min** + 체결 델타 | 근사 실시간이면 충분 | WS 불필요 |
| **비활성 계좌 지연계산** | **읽을 때만(lazy)** | 온디맨드 | 아래 캡 IC-5 |

- **캡 IC-5(비활성 지연계산):** 최근 7일 세션·미체결 대기주문·포지션 변동이 없는 계좌는 **주기 평가 대상에서 제외**하고 순위·NAV를 **조회 시점에만** 계산한다. → 대회 참가자 수가 아니라 **활성 참가자 수**에 평가비가 비례. 20k 참가 대회에서 활성 8종목 보유자 기준 10s 재계산 = 20k×8 = 160k 캐시 마크리드/10s(=16k/s) + 2만 스코어 정렬(<10ms) → **컴퓨트는 사실상 무료, egress가 유일한 스케일 변수**임을 확인.

#### 12.6.4 데이터 과금 곡선 — 2개 시나리오 + per-subscriber 트랩

데이터 비용은 **과금모델에 따라 MAU와의 관계가 근본적으로 다르다.**

**시나리오 A — 호출량 과금(Polygon/TwelveData 재배포 플랜):** 백엔드 팬인(동일 종목 다수 유저 = 업스트림 1콜, MKT-07)으로 **비용이 MAU와 분리**된다. 상한은 `distinct_hot_symbols ÷ cache_TTL`이며 유저 수와 무관.
- 예: 핫심볼 ~200 × 캐시 TTL 2~5s → 업스트림 40~100 calls/s 수준, 재배포 실시간 플랜(예: Polygon Advanced $199/월, 추정) **평탄 정액**. MAU가 10k→500k로 늘어도 데이터 청구는 그대로.

**시나리오 B — per-subscriber/per-user 과금(트랩):** 비용 = 단가 × 구독자(≈MAU). 무과금 90% 베이스에 치명적.
- 미국주식 **SIP 통합테이프(CTA/UTP) per-subscriber 피**가 대표 위험. 벤더가 거래소 non-professional 구독자 피를 **패스스루**하면 유저당 월 ~$1~3(디스플레이 non-pro, 추정), professional·depth는 수십 배. **150k MAU × $1.5 = 월 $225k** → §12.4 매출을 8배 초과, **영구 적자**.
- Polygon류 '평탄 재배포 플랜'이 안전한 이유가 바로 이것: 벤더가 거래소에 business-line 라이선스로 정산하고 우리는 **유저 수 무관 정액**을 낸다. 반대로 **거래소 직결/SIP 직접 구독/per-subscriber 패스스루 벤더**는 게이트에서 차단한다.

- **계약 게이트 IC-6(per-user 금지):** 무과금 소비자 베이스 대상 **실시간 주식 데이터 계약은 per-subscriber/per-user 과금이면 서명 금지.** 허용: (a) 호출량·정액 재배포 플랜, (b) 지연/일봉, (c) 크립토 거래소 WS($0). 계약서에 **캐싱 TTL ≥ 우리 푸시 주기(N초)** 명문화(MKT-07 캐시 정책과 충돌 방지), attribution·재배포 허용·해지 조항 확인 후에만 진행(MKT-06 체크리스트에 본 조항 추가).

#### 12.6.5 손익분기 유저 수 (호출량 모델 vs 트랩)

블렌디드 net ARPU **$0.116/MAU/월(기본 시나리오, §12.4)**, 동접(경쟁)을 MAU의 ~3%로 가정(추정).

| MAU | 동접(≈3%) | **A. 호출량모델 비용/월** | B. per-user 트랩 비용/월 | net 매출/월($0.116) | A COGS/매출 | 판정 |
|---|---|---|---|---|---|---|
| 10k | ~300 | 데이터$199 + 인프라~$300 ≈ **$500** | $15,000 | $1,160 | 43% | A: 흑자 / B: 적자 |
| 50k | ~1.5k | $199 + ~$480 ≈ **$680** | $75,000 | $5,800 | 12% | A: 흑자 / B: 적자 |
| **150k** | ~4.5k | $199 + ~$1,300 ≈ **$1,500** | $225,000 | $17,400 | 9% | A: 흑자 / **B: 대규모 적자** |
| 500k | ~15k | $199 + ~$3,600 ≈ **$3,800** | $750,000 | $58,000 | 7% | A: 흑자 / B: 적자 |

- **손익분기:** 호출량 모델에서 데이터+인프라 COGS는 **매출의 5~9%**로 수렴(수천~수만 MAU 부근에서 흑자 전환) → §12.4의 "관리 가능"은 **A 시나리오 한정 참**. per-user 트랩은 **어떤 MAU에서도 손익분기 없음**.
- **CCU 설계상한 정합(NFR-S02):** 설계 목표 CCU는 **단일값 10k**(NFR-S02)로 고정한다. 위 표의 **500k MAU × ~3% ≈ 15k 동접**은 이 10k 설계상한을 **초과**하므로, 500k MAU 규모는 **워커 수평확장·샤딩을 선행 통과한 뒤에만** 진입한다(용량초과 구간, §10.3.3 로드게이트 G1~G5). 즉 10k는 "무증설로 보장하는 상한"이고, 그 이상은 수평확장 경로(NFR-S01)로만 대응한다.
- 한국 실시간(KRX 정보사용료+재배포)은 별도 견적(연 수백만원대 가능성, 추정)으로 위 표 밖 → 국내장은 지연/일봉으로 시작(MKT-06 엣지케이스 정합).

#### 12.6.6 유닛 이코노믹스 편입 (신규 COGS 라인)

§12.4 유닛 이코노믹스에 **인프라·데이터 COGS/MAU** 라인을 신설한다.

- **인프라·데이터 COGS/MAU(추정):** 150k MAU 기준 월 ~$1,500 ÷ 150k = **~$0.010/MAU/월** (기본 net ARPU $0.116의 ~9%, IC-7 10% 게이트 이내).
- **게이트 규칙 IC-7:** 어떤 실데이터/실시간 인프라 결정도 **COGS/MAU가 블렌디드 ARPU의 10%를 초과하면 승인 게이트에서 반려**한다(광고 ARPU 재검토 리스크 §16 반영 시 임계 자동 하향). per-user 데이터는 이 게이트를 구조적으로 통과 불가 → 계약 단계(IC-6)에서 선차단.
- **핵심 정합:** §12.4 "진짜 고정비 리스크는 UA·백엔드 인프라"를 정량화하면 — UA는 여전히 최대 병목이나, **백엔드 인프라 리스크는 'MAU'가 아니라 '경쟁모드 동접'에 비례**하며(솔로=0), 캡 IC-1~5로 상한이 걸리고, **데이터 리스크는 과금모델 선택(IC-6)으로 이분법적으로 결정**된다. 즉 리스크는 '규모'가 아니라 **모드 분리·캡·계약 3개 레버**로 제어 가능하다.

### 16.3 놓치기 쉬운 2차효과 (감사 요약)
- 하이롤러 격리 리그 존재 자체가 심사 리스크 → **출시 초기 스코프 제외**, 유저풀·평판 안정 후 옵트인·성인전용·비현금·강한 면책·심사노트 첨부.
- 위험조정 리더보드가 새 어뷰징(지표 게이밍)을 학습시킴 → 기간 일관성·규칙준수율 가중.
- 실데이터 신뢰 향상이 '실전처럼 느껴짐→실전 이식·자문 오인' 규제 노출 증가 역설 → 면책·"연습 전용" 프레이밍 유지.
- 다계좌×2초tick×종목수 성능 곱증가, 대회 리더보드는 비활성 계좌도 서버 스냅샷 요구(지연계산 상충) → 뷰포트 티어링+서버측 집계.
- 실패 불가 첫 매수가 매수를 trivialize(Robinhood 경계 지점) → '배움 축하' 프레이밍 + 직후 리스크 균형 카드.
- 배지 중첩(지연·모드·면책·출처)이 정보 위계 붕괴 → 배지 시각 예산 관리.

## 19. 비기능 요구·릴리스 기준

> **원칙:** NFR은 "빠르면 좋다"가 아니라 **숫자·측정도구·판정주체·릴리스 연결**을 가진 계약이다. 각 항목은 (1)정량 목표, (2)측정 방법·표본, (3)위반 시 조치(릴리스 차단/경보/부채등록)를 명시한다. **G=하드 게이트(미달 시 해당 단계 릴리스 차단)**, **W=경보(부채 등록·다음 스프린트 시정)**. 측정은 자동화(CI·대시보드) 우선, 수동 QA는 보조. 릴리스 게이트 원칙과 단계별 종료 기준(Exit Criteria)은 §19.7~19.8에 함께 둔다.

### 19.7 릴리스 게이트 원칙 + Phase 0 차단 게이트

> **원칙:** 각 마일스톤은 아래 **Exit Criteria 표(§19.8)를 100% 충족(모든 G 항목 PASS)해야 다음 단계로 진입/출시**한다. Exit Criteria는 "했다"가 아니라 **측정치·판정주체·증빙(대시보드/CI 링크)**을 요구하며, PM이 릴리스 회의에서 항목별 PASS/FAIL을 기록한다. FAIL 1건이라도 있으면 릴리스 차단(예외는 O-이슈로 등록 후 서면 승인).

#### 19.7.1 Phase 0 완료 = 하드 릴리스 차단 게이트 (모든 이후 단계의 선행)

Phase 0(§6.1 회계·엔진 하드닝)은 **UX 변화 0의 정확성 계약**이며, **미완료 시 MVP를 포함한 어떤 빌드도 스토어·베타로 내보내지 않는다.** "병렬 착수 유혹 차단"(§16.1)을 게이트로 강제한다.
